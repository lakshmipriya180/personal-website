01 / The Problem

The invisible workload no app was built for

Dual-income couples with children run their households on a fragile stack of shared Google Calendars, WhatsApp threads, and memory. One partner — statistically the woman in 80% of households — silently carries the cognitive overhead: booking appointments, remembering school events, planning meals, tracking kids' activities.

Her partner can't see this work. And because he can't see it, he can't share it.

Before Ouralink
School newsletter arrives. One parent reads it, makes a mental note, reminds the other three times over WhatsApp, manually creates a calendar event, and adds a grocery item — all separate actions across four apps. The cognitive load of doing this for every email, every week, for every child, is invisible and unmeasured.
After Ouralink
Parent forwards the email to the household address. Within 30 seconds, Claude has extracted the event, dated it, and classified it as a planning task. One tap to confirm. The calendar event is created. The other partner is notified. The load is logged. The email is gone — never stored.
02 / Architecture Decision

Why I moved all AI calls to a dedicated server

The first version of Ouralink ran Claude API calls inside Supabase edge functions. They kept timing out. After investigating, I found the root cause: Supabase's internal XHR wrapper has a hard timeout that cuts off long-running AI calls before they complete. There was no configuration to extend it.

The fix wasn't a tweak — it was a rearchitecture. I moved every AI call to a dedicated Node.js server on Railway. This decision unlocked streaming responses, proper retry logic, and a single secure location for the API key.

The API key rule — enforced by CI/CD
ANTHROPIC_API_KEY exists in exactly one place: Railway environment variables. A grep check runs on every push to main. If the key pattern appears anywhere in the mobile or web codebase, the build fails. Not a code review culture — an automated enforcement.

03 / Trust Design

Every AI output is steerable, not final

Trust design in Ouralink means: AI proposes, human disposes. No AI output creates a permanent change without a human confirmation step. This is consistent across every feature — not because of policy, but because of architecture.

"The confidence bar on a parsed email isn't decoration. It tells the parent whether to trust the extraction or read it carefully."

Confidence signals

Every parsed email has a Claude-generated confidence score (0.0 – 1.0) shown as a proportional bar. Low confidence (<0.6) surfaces an amber chip: "Check this one." The parent sees what the AI thinks — and how certain it is.

User steering

Parsed items can be accepted as a task or as a calendar event — the user can override Claude's classification. Date ideas show the AI reasoning below each suggestion: "Light week — good for a longer afternoon." Meal plans are editable per cell, tracked as manual_override in the database.

Safe failure modes

When meal generation fails to parse, the app shows a designed empty state with retry and manual entry — not a crash. When email parsing produces no clear action, confidence is set to 0.1 and the amber chip prompts review. Failure is graceful and informative.

Non-clinical language

The mental load tracker shows bars — never percentages, never numbers. The mood signal uses 💡 not ⚠️. Overdue states use amber not red. A CI/CD check enforces that banned words (burnout, overwhelmed, stressed, failed) never reach the UI. Language is trust design too.
04 / Production Reliability

The bugs that only appear in production

Three reliability problems found in production that no demo or test environment would have surfaced. Each one has a measurable before/after.

60%→100%
Signup reliability — the RLS cold-start bug New user signs up. The app immediately reads their household record back from the database. During the RLS policy cold-start window (<200ms), the row exists but the policies haven't loaded yet — the read returns null. The user sees an error and abandons. Fixed by adding a 300ms retry on the post-signup read with exponential backoff. Signup reliability: 60% → 100%.
0→6
RLS policy audit — six tables using the wrong security function Audited all database security policies and found six tables using inline auth.uid() checks instead of the correct SECURITY DEFINER function. The result was silent auth failures — requests failing with no error surfaced to the user. Bulk-fixed all six, then wrote a new-table checklist so the same mistake cannot happen again.
Race → Fixed
Reward redemption race condition A parent approves a reward. Between the approval request and the point deduction, a second approval fires. Both succeed — balance goes negative. The balance check was only at request time, not at approval time. Fixed by re-checking the balance inside the approval transaction — a guard that costs one extra query and prevents a user-visible integrity failure.

The three iron rules — non-negotiable, enforced automatically

Rule 1 — Email content never stored

Raw email text lives in one local variable. It is passed to Claude and cleared in a finally block — even if Claude throws. Only the structured extraction (title, type, date, confidence) enters the database. Verified by CI/CD grep on every commit.

Rule 2 — AI key in one place only

ANTHROPIC_API_KEY exists only in Railway environment variables. Never in mobile bundles (extractable). Never in Next.js client. Enforced by automated grep check on every push to main.

Rule 3 — Points ledger is immutable

kid_point_ledger is INSERT-only. No UPDATE, no DELETE anywhere in the codebase. Balance = SUM at query time. Enforced by grep check. A child's earned points cannot be silently erased by a bug.

05 / Performance

Perceived speed is a design decision

AI responses are inherently slow. A 4-second wait before anything appears feels broken. Streaming changes the experience: the first idea appears in under 1 second, the rest follow. The user starts reading while the model is still generating.

Without streaming
User taps "Suggest date ideas." Spinner for 4.2 seconds. All 5 ideas appear at once. Feels like waiting for a slow server. Drop-off rate on this feature: high.
With SSE streaming
First idea appears in 0.8 seconds. User starts reading idea 1 while ideas 2–5 stream in. Perceived wait: under 1 second. Same model. Same latency. Different experience. 
// Railway API — stream Claude response as Server-Sent Events
const stream = await anthropic.messages.stream({
  model: 'claude-3-haiku-20240307',
  messages: [{ role: 'user', content: prompt }],
  max_tokens: 1000
})

for await (const chunk of stream) {
  // Send each idea as it completes — don't wait for all 5

06 / Instrumentation

Measuring whether the AI is actually working

Accept rate on parsed emails is the live proxy for Claude's extraction quality. If accept rate drops below 60%, the prompt needs iteration. This metric requires no ML infrastructure — it's a ratio stored in the same database the app already uses.

07 / Human-in-the-Loop

AI proposes. Humans decide. Always.

Ouralink uses a consistent propose-confirm pattern across every AI feature. This is a named design pattern — not accidental. No AI action in Ouralink creates a permanent change without a human confirmation step.

✉️ Email
Claude extracts task or event → parent reviews → one tap confirms. Confidence bar shows certainty level. Type can be overridden. Nothing auto-creates.
💡 Ideas
Claude generates date ideas → parent browses → explicit schedule tap. AI reason shown on each card. Wishlist saves ideas without committing to them.
🍽 Meals
Claude generates weekly meal plan → parent can edit any cell → saved as manual_override. Grocery list generated from confirmed plan only.
⭐ Rewards
Kid requests reward → parent receives notification → parent approves or denies. Balance re-checked at approval time (race condition guard). Points only deduct on explicit approval.
